# P5 — 堅牢化: 公開に耐える配管

> ロードマップ `docs/ROADMAP.md` の P5 節を実装単位に落とした指示書。
> **目的**: 不特定多数に触られても壊れない・赤字にならない状態。

## スコープ(この PR でやること)

1. **クォータの実ブロック(429)**
   - 月間上限(`FREE_MONTHLY_QUOTA`、既定 100)を超えたら `429 {error:"quota_exceeded", limit, used}` を返す
   - 対象は「新しい問い」= `/api/analyze` と `/api/analyze/stream`。深化・共鳴は同じ問いの掘り下げなのでカウント対象にはしないが、上限超過中は同様にブロックする(コスト防衛)
   - カウントは従来どおり成功後に `incrementQuota`。チェックは実行前に読むだけ(KV結果整合のため多少の超過は許容 = ユーザーを不当にブロックしない安全側)

2. **連打防止(anti-hammer)**
   - 同一 `clientId` の **同時実行1本**(best-effort ロック)+ **最小リクエスト間隔**(既定 1500ms)
   - KV キー `rl:{clientId}` に `{inFlight, lockUntil, last}` を保存(TTL短め)
   - 同時実行中 → `429 {error:"in_progress"}`、間隔不足 → `429 {error:"too_frequent", retryAfterMs}`
   - 全4エンドポイント(analyze / stream / deepen / resonate)に適用
   - KV は結果整合で厳密な相互排他は不可。あくまで軽量な連打抑止(設計メモ参照)

3. **全体タイムアウト予算**
   - リクエスト全体の上限 `REQUEST_BUDGET_MS`(既定 30000)。`AbortSignal.timeout` を deps 経由で
     `runNodes / synthesize / verify`(既に signal 対応済み)へ流し、超過で中断
   - Workers 無料プランの CPU 時間制限は「fetch 待機は CPU を消費しない」ため 8 並列でも問題ない。
     ここで制限しているのは**実時間**の暴走であり CPU ではない(意識のための明記)

4. **入力バリデーション再点検 + 異常系テスト**
   - `input` 4000字 / `summary` 500字 / `priorAnswer` 8000字は実装済み。境界・型・欠落の異常系テストを追加

5. **運用スクリプト `ops/report.ts`**
   - KV の `cost:{yyyymmdd}:*` を集計し、日次の **総コスト・件数・失敗率(fallback率)・平均ms** を表で出力
   - `npm run ops:report`(既定は当日、`--days N` で遡及、`--json` で機械可読)
   - KV アクセスは `wrangler kv` 相当。テストは純関数(集計ロジック)を対象にモックで検証

## 設計メモ

- **summary 改ざん**: `summary` はクライアント経由のローリング要約。改ざんしても悪化するのは**本人の回答品質だけ**でありサーバー資源・他者への影響はない(リスク低)。サーバーは長さのみ検証する。
- **クォータの厳密性**: KV は結果整合。バースト時に上限を数件超える可能性があるが、同時実行ロック+最小間隔が第一防壁として効くため、コストが**線形に増えない**目標は満たす。厳密な原子性が必要になった段階で Durable Objects を検討(将来)。
- **第二防壁(任意・将来)**: 日次コスト上限の全体ブレーカー(超過で全リクエストを simple に落とす)は本 PR では入れない。ROADMAP に将来項として残す。

## 受け入れ条件

- [ ] 上限超過で `/api/analyze(/stream)` が 429 を返し、UI にメッセージが出る
- [ ] 連打(同時・高頻度)が 429 で弾かれ、コストが線形に増えない
- [ ] 全体予算超過でリクエストが中断される(モデル呼び出しが打ち切られる)
- [ ] 入力異常系(空・超過・型違い・欠落)のテストが緑
- [ ] `npm run ops:report` が日次表を出力する
- [ ] `tsc` / `vitest`(既存+新規)緑、web ビルド成功
