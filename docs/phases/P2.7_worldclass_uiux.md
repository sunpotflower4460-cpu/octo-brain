## 0. このフェーズの結論

OctoBrainを「タコが背景で動くサイバー風チャット」から、**8本の腕が何をして、どの視点が集まり、中央脳がどう一つの答えへ織っているのかを直感的に理解できる思考インターフェース**へ進化させる。

未来感はネオンの量では作らない。

- 内部の因果が見える
- 動きに意味がある
- 情報の階層が静かで明快
- 待ち時間が「処理中」ではなく「知性が立ち上がる時間」に変わる
- ユーザーが深化・共鳴へ自然に介入できる

この5点で作る。

プロダクトの中心コピーは次とする。

> **8つのレンズで観て、4つの軸で張り合い、1つの理解へ織る。**

デザイン上の中心思想は次とする。

> **タコを壁紙から、思考の計器へ。**

## 1. 現状診断

現行実装には、すでに強い独自性がある。

- 8レンズの並列処理
- 4つの緊張軸
- 中央脳による統合
- SSEストリーミング
- TENSIONの深化
- RESONANCEの掛け算
- 腕の発光による状態演出

一方、UIではこの構造の多くが十分に伝わっていない。

### 1.1 起きていることが二段階に圧縮されすぎている

バックエンドは `routing → nodes → synth → verify` の4段階だが、画面では主に「並列解析中」「中央脳が統合中」の2種類しか見えない。ユーザーは、今どこまで進んだか、何本の腕が完了したか、検証中なのかを把握できない。

### 1.2 タコが機能ではなく背景になっている

`OctopusCanvas` は美しいが、全面背景のためチャット本文と視覚的に競合する。腕とレンズ名・完了状態の対応も画面上で結び付かず、「なぜ光ったのか」が初見では分からない。

### 1.3 重要度の異なる情報が同じ密度で縦に積まれる

回答、緊張、共鳴、8視点、共鳴結果、メタ情報が同じ会話列に追加されるため、機能が増えるほど回答の主役が埋もれる。`plan / domain / quorum / ms` も一般ユーザーには説明なしで露出している。

### 1.4 モバイルで窮屈になりやすい

- `absolute` な入力フッター
- `min-h-screen` と `overflow-hidden`
- 1行の `input`
- 2列の小さな視点カード
- 9〜11pxの補助文字
- Safe Area・ソフトウェアキーボードへの未対応

この組み合わせは、iPhoneで情報の重なり・誤タップ・読みにくさを生みやすい。

### 1.5 待機・中断・失敗のUXが弱い

- 実行を停止できない
- SSEエラー時に再試行しにくい
- HTTPの詳細がそのままユーザー文面に混ざる
- トークン到着ごとの自動スクロールで、過去回答を読んでいる人を最下部へ引き戻し得る
- 全画面Canvasがアイドル中も常時描画される

### 1.6 アクセシビリティと動きの選択肢が不足している

進行状態の `aria-live` 通知、CanvasのDOM代替、44px以上のタップ領域、色以外の状態表現、`prefers-reduced-motion` への対応が不足している。

## 2. 成功条件（North Star）

初めて触る人が、説明書なしで次のことを理解できる状態を完成とする。

1. OctoBrainは普通の1モデルのチャットではない
2. ライトとディープで使う腕の数が違う
3. 今、問いの分類・レンズ分析・統合・検証のどこにいるか
4. どのレンズが完了・失敗・未起動なのか
5. 最終回答が主役で、詳細は必要なときだけ見られる
6. 「深く掘る」と「掛け合わせる」の違い
7. 処理を停止・再試行できる

### 体験の品質基準

- **5秒理解**: 初回画面を5秒見れば「8つの視点を統合するAI」と分かる
- **待ち時間の透明性**: どの段階か、完了レンズ数はいくつかが常に分かる
- **回答優先**: 回答本文の読みやすさが装飾に負けない
- **段階的開示**: 技術情報は隠さず、初期状態では畳む
- **未来感と温度**: 冷たい管理画面ではなく、生物発光する知性のように感じる
- **モバイル優先**: 390px幅のiPhoneで最も気持ちよく使える

## 4. 守るべき製品原則（それ以外の実装制限は外す）

### 4.1 思考過程を生で見せない

画面に出すのは以下だけ。

- フェーズ名
- 進捗状態
- APIがすでに返している短い `opinions`
- TENSION / RESONANCEの要約
- 最終回答

隠れた推論、内部プロンプト、逐語的なChain of Thoughtは表示しない。「透明性」は内部思考の暴露ではなく、**処理構造と結果の透明性**で実現する。

### 4.2 嘘の進捗率を表示しない

LLMの終了時刻を正確に予測できないため、`72%` のような架空のパーセンテージや残り秒数は出さない。代わりに以下を表示する。

- 現在フェーズ（1/4〜4/4）
- 完了レンズ数（例: 5/8）
- 各レンズの状態
- 経過時間（必要なら詳細内）

### 4.3 回答本文を最優先する

ネオン、グロー、Canvas、カード枠は回答の可読性を落としてはならない。文字の背後では強い発光や高速な動きを止める。

### 4.4 機能の意味を壊さず、実装は大胆に変えてよい

既存の `/api/analyze/stream`、`/api/deepen`、`/api/resonate` と型は、最高の体験に必要なら協調的に変更してよい。未公開プロダクトのため、後方互換を守ること自体を目的にしない。フロント・Workers・テスト・ドキュメントを同じPRで更新し、既存機能の同等以上の動作を保証する。

モデル選定・プロンプト・分析ロジックも「絶対に触るな」とはしない。ただしUIを成立させるために必要な変更だけに限定し、回答品質・原価・キャッシュ・安全性への影響を検証する。新しいモデル呼び出しを追加する場合は、`callModel` と原価ログを必ず通し、なぜローカル処理では実現できないかを完了報告に書く。

### 4.5 `OctopusCanvas` の既存保護ルールを撤廃する

現行 `CLAUDE.md` の「描画ロジックの挙動変更禁止」は、このフェーズでは撤廃する。実装前にルール4を次の趣旨へ更新する。

> OctoBrainの生体的な中枢、8レンズと8腕の対応、4軸、深化と共鳴の意味を体験として保持する。現行Canvasの数式・見た目・props・レンダラー・ファイル構成を保持する義務はない。Canvas 2D、SVG、DOM、WebGL、Three.jsまたはハイブリッドから最適な方法を選び、全面再設計してよい。アクセシビリティ・モバイル性能・情報理解を装飾より優先する。

元の触手計算式を残すか捨てるかは、完成度で判断する。単に過去コードを守るために妥協しない。守るのはコードではなく「分散した腕が感じ、中央へ戻り、必要なら腕同士が張り合い・響き合う」という体験の意味である。

## 5. 新しい体験構造

### 5.1 画面の3層

画面を次の3層に整理する。

1. **Living Core（生きた中枢）**  
   タコ、8レンズ、現在フェーズ、完了数を表示する機能的な思考計器
2. **Conversation（対話）**  
   ユーザー入力と最終回答。最も広く、最も静かに読む場所
3. **Composer（問いの入口）**  
   モード選択、複数行入力、送信/停止

### 5.2 デスクトップ

`min-width: 1180px` 以上では2カラム。

- 左: Living Core（幅320〜360px、sticky）
- 右: Conversation（本文最大幅760px）
- 全体最大幅: 1180〜1240px
- ComposerはConversation列の下部に固定/追従

タコを全面背景から左の専用ステージへ移す。背景には非常に薄い海中粒子・放射状グラデーションだけを残してよい。

### 5.3 タブレット

1カラム。Living Coreを会話上部の横長パネルにする。8レンズは4×2または横スクロール。処理完了後はコンパクト表示へ縮められる。

### 5.4 モバイル

- `100dvh` を使用
- 上部に高さ120〜170pxのCompact Core
- 処理中はCoreを表示、完了後は56〜72pxのステータスバーへ縮小可能
- Conversationは1列
- ComposerはSafe Area込みで下部にsticky/fixed
- Node詳細はインラインの2列カードではなく、1列リストまたはBottom Sheet/Drawer
- 320px幅でも横スクロールを発生させない

### 5.5 初回の空状態

初回は長い起動メッセージをチャット吹き出しにしない。専用Heroにする。

表示例:

**OctoBrain**  
8つのレンズで観て、4つの軸で張り合い、1つの理解へ織る。

短い補足:

> 決断、アイデア、本音の整理を、ひとつのAIでは見落とす角度から考えます。

入力例チップを3つだけ置く。

- 決断を整理する
- アイデアを広げる
- 本音を深く見る

チップは入力欄へ例文を入れるだけで、自動送信しない。

### 5.6 初回オンボーディング

初回起動時だけ、スキップ可能な3場面の短いオンボーディングを表示する。説明スライドを読む体験ではなく、Living Coreが実際に反応するインタラクティブな導入にする。

1. **8本の腕** — それぞれが違う角度から同時に観る
2. **4つの軸** — 反対側の視点が張り合う場所に深さが生まれる
3. **中央脳** — 一致も矛盾も消さず、ひとつの理解へ織る

要件:

- 1場面あたり1文 + 1つの動き
- `次へ`、`スキップ`、最終の`問いを置いてみる`
- キーボード・スクリーンリーダー対応
- Reduced Motionでは静的な図へ
- 完了状態はローカル保存し、設定から再表示可能
- オンボーディングを閉じないと使えない強制壁にしない

### 5.7 会話の継続性

P6まで先送りせず、今回ローカル会話永続化を実装する。サーバーアカウントは不要。

- IndexedDBを基本とし、Storage Adapterを介して実装
- 会話、メッセージ、ThoughtTraceの完了結果、summary、作成/更新日時を保存
- SSE処理中の一時状態は、再起動時に`中断されました`へ正規化
- 新規会話
- 会話一覧
- 会話名の変更
- 削除（確認あり、Undo可能なら望ましい）
- 最終ユーザー入力からローカルで短い初期タイトルを作る。タイトルだけのためのLLM呼び出しは不要
- 各会話の入力ドラフトを自動保存
- デスクトップは左Drawer/Sidebar、モバイルはSheet
- 検索はローカル実装が軽量なら追加してよい

保存失敗やプライベートブラウズでもチャット本体を止めない。メモリ内へフォールバックし、非致命通知を出す。

### 5.8 設定

最低限、次を提供する。

- Motion: システムに従う / 少なく / 通常
- Living Core: 自動縮小 / 常に表示
- 処理詳細: 標準 / 詳細
- オンボーディングをもう一度見る
- ローカル会話データを削除

設定画面そのものも世界観を壊さないが、装飾より明快さを優先する。設定値はローカル保存する。

## 6. 思考進行UI（このフェーズの核心）

### 6.1 UI状態を9状態に拡張する

表示用状態を次の判別可能なUnionまたはReducerで管理する。

```ts
type UiPhase =
  | "idle"
  | "routing"
  | "nodes"
  | "synth"
  | "verify"
  | "deepening"
  | "resonating"
  | "done"
  | "error"
  | "cancelled";
```

Canvas用の既存 `AppState` は派生値として残す。

- `routing | nodes | resonating` → `processing_subs`
- `synth | verify | deepening` → `processing_main`
- その他 → `idle`

グローバルな3状態に全UIを押し込めない。各回答メッセージが自分の `trace` を持つ構造にする。

### 6.2 フェーズ名とユーザー向け文言

| phase | 表示名 | 補足文 |
|---|---|---|
| routing | 問いを読む | どの軸とレンズで考えるかを選んでいます |
| nodes | 8つのレンズ | 複数の腕が同時に異なる角度を観ています |
| synth | 視点を織る | 中央脳が一致と緊張をひとつの理解にまとめています |
| verify | 光を整える | 矛盾・断定の強さ・安全性を最終確認しています |
| done | 結晶化 | 回答がまとまりました |
| deepening | 緊張を深く掘る | 対角の2本が互いの意見を読み直しています |
| resonating | 遠い視点を響かせる | 2本の腕から第三の選択肢を探しています |

「解析」「処理」「ニューラルネットワーク同期」の連発を避け、世界観は保ちながら日本語として意味が分かる文にする。

### 6.3 Process Dock

各ユーザー入力の直後、回答が始まる位置に `ProcessDock` を表示する。

構成:

- 上段: 現在フェーズ名 + 4段階のステップ表示
- 中段: 8レンズ（ライトなら起動4、ディープなら8）
- 下段: `完了 5/8`、縮小/詳細ボタン、停止ボタン

処理完了後は高さを縮め、`8視点で思考済み · 6.2秒` のような1行Disclosureへ変わる。ユーザーが開けばレンズ結果を確認できる。

### 6.4 レンズ状態

各レンズに次の状態を持たせる。

```ts
type LensUiStatus =
  | "inactive"
  | "queued"
  | "working"
  | "done"
  | "timeout"
  | "parse_error"
  | "error";
```

状態表現は色だけに頼らない。

- inactive: 「今回は未使用」・細い輪郭
- queued: 小さな点
- working: ゆっくりした光の移動
- done: チェック + そのレンズ固有の柔らかな発光
- timeout: 時計アイコン + 「時間切れ」
- parse_error: 壊れた波形アイコン + 「読み取り失敗」
- error: 警告アイコン + 「接続失敗」

失敗ノードは消さないが、恐怖を煽る赤一色にもしない。「一部の腕は返答できませんでしたが、残りで統合しました」と全体へ影響を説明する。

### 6.5 8レンズのDOM可視化

Canvasだけに意味を背負わせず、スクリーンリーダー・Reduced Motion・小画面でも分かるDOMの `LensOrbit` を作る。

- デスクトップ: 中央Coreの周囲または下部に8レンズを円環/4軸配置
- モバイル: 2×4のコンパクトグリッドまたは横スクロールしない4×2
- `nodeDisplay.ts` の順序・UI名・絵文字を唯一の正とする
- 対角軸を細い線またはペアラベルで示す
- Raw ID（`reason`, `values` 等）は画面に出さない

装飾図にせず、クリック/タップでそのレンズの短い意見を開ける実用UIにする。

### 6.6 ストリーミング回答

- 回答開始前は骨組みスケルトン1個だけ。複数の点滅要素を置かない
- 最初のtoken到着で回答カードへ自然に遷移
- tokenごとに全メッセージを再描画しない。バッファし、`requestAnimationFrame` または最大30fpsでまとめて反映
- ストリーミング中のカーソルは細く静かに点滅
- `verify` 中は回答を読めるままにし、カード上部に小さく「最終確認中」
- `done.answer` がVerifierで差し替わった場合、内容を瞬間的に入れ替えず短いクロスフェードを使う

## 7. 最終回答の情報設計

### 7.1 吹き出しからInsight Cardへ

ユーザー入力は短い吹き出しでよい。OctoBrainの回答は通常のチャット吹き出しではなく、静かな `InsightCard` として表示する。

- 読書幅: 60〜72文字程度
- 本文: 15〜17px、line-height 1.75前後
- 長文の段落間余白を確保
- 背景は高コントラストの不透明寄りSurface
- 強いグローを本文背面に置かない

### 7.2 安全なMarkdown表示

回答は `react-markdown` + `remark-gfm` 等で安全に表示する。生HTMLは許可しない。以下を整える。

- 見出し
- 箇条書き・番号付きリスト
- 引用
- インラインコード・コードブロック
- 表（モバイルでは横スクロール領域内）
- リンク（外部であることを示す）

### 7.3 回答アクション

回答下部に、常時目立たせすぎないアクションを置く。

- コピー
- 共有（Web Share API、非対応ならコピーへフォールバック）
- 再試行
- 思考の詳細

アイコンだけにせず、少なくとも初回またはモバイルでは短いラベルを付ける。

### 7.4 メタ情報の段階的開示

`plan / domain / quorum / ms / cost` は本文直下へ生で並べない。

`処理の詳細` Disclosure内で人間向けに変換する。

- 使用モード: ライト / ディープ
- 観点: 4 / 8
- 統合に使えた腕: 例 7 / 8
- 応答時間: 6.2秒
- 検証: そのまま / 表現を調整
- 開発ビルドのみ推定原価

`domain` は「今回の問い: 仕事」のように必要な場合だけ表示する。

## 8. TENSIONとRESONANCEの再設計

### 8.1 2つを「次の思考」領域にまとめる

回答下に `次に、どう考える？` セクションを作り、その中へ2種類のカードを置く。

#### TENSION — 深化

- 色: アメジスト〜マゼンタ
- 意味: 対立・張力・まだ決め切れていない場所
- 見出し例: `心の軸に、まだ張りがあります`
- 軸の両端のレンズを明示
- CTA: `この緊張を深く掘る`

#### RESONANCE — 共鳴

- 色: シアン〜乳白
- 意味: 遠い2視点の共通根・第三の選択肢
- 2つの意見を視覚的な線/結び目で接続
- `root` を「共通する根」として表示
- CTA: `2つを響かせる`

### 8.2 優先順位

両方ある場合も同じ強さの大ボタンを2個並べない。

- AIが特に強く検出した方をPrimary
- もう一方はSecondary
- どちらも実行済みなら結果をカード内に折り畳んで保持

### 8.3 ユーザー選択の共鳴

「8つの視点」詳細で意見を2つ選ぶ操作を改善する。

- 各意見のタップ領域は44px以上
- 選択中はチェック、番号 `1` / `2`、レンズ名を表示
- 画面下または詳細パネル下に選択バー `2つ選択中` を固定
- 同じレンズを選んだ場合、ボタンを黙って無効化せず理由をその場で示す
- 3つ目を選んだ場合のFIFO入れ替えは、短いトーストで通知
- キャンセル/選択解除を明示

## 9. Composer（入力体験）

### 9.1 複数行入力

`input type="text"` を自動伸長する `textarea` に変更する。

- 1〜6行で伸長
- デスクトップ: Enterで送信、Shift+Enterで改行
- IME変換確定中は送信しない（`isComposing` を考慮）
- モバイルでは送信ボタンを確実に使える
- 4000字上限へ近づいた場合のみ文字数表示

### 9.2 モード選択

現在の `ライト (2軸4腕)` / `ディープ (4軸8腕)` は、初見には技術的すぎる。

表示を次のようにする。

- **ライト** — 4つのレンズで素早く整理
- **ディープ** — 8つすべてで深く考える

選択UIは小さなSegmented Controlにし、説明はツールチップ/Disclosureへ。処理中は変更不可。将来の有料化を示すロックや価格は今回入れない。

### 9.3 送信中

送信ボタンを停止ボタンへ変える。`AbortController` を実装し、ユーザーがSSEを中断できるようにする。

中断後:

- 入力は失わない
- 途中までの回答は残す
- `生成を停止しました` と表示
- `続きを再試行` または `最初から再試行` を選べる

### 9.4 Safe Areaとキーボード

- `padding-bottom: env(safe-area-inset-bottom)`
- `100vh` ではなく `100dvh`
- Composerの実高に応じてConversation下余白を動的に確保
- iOSキーボード表示で入力欄が画面外へ押し出されない
- body全体の不自然な二重スクロールを作らない

## 10. 自動スクロール

現在の「messagesまたはappStateが変わるたび最下部へsmooth scroll」を廃止する。

## 12. Motion Design

Motionは装飾ではなく状態変化を説明する。

### 12.1 基本時間

- hover/press: 100〜140ms
- 小パネル: 180〜240ms
- Coreの状態遷移: 360〜600ms
- アイドル呼吸: 4〜7秒周期、振幅は小さく

### 12.2 フェーズ別の意味

- routing: Core中心に細い輪が1〜2回集まる
- nodes: 中央から腕先へ光が流れ、完了した腕だけ静かに定着
- synth: 腕先から中央へ光が戻る
- verify: 中央にゴールドの細いリングが一周
- done: 1回だけ穏やかに発光して静止状態へ
- deepening: 対角2本が交互に脈打つ
- resonating: 離れた2本の位相が揃う

常時 `animate-pulse` を複数箇所で重ねない。

### 12.3 Reduced Motion

`prefers-reduced-motion: reduce` の場合:

- 触手の振幅・移動速度を大幅に下げるか静止画相当へ
- スライド・拡大縮小をフェードへ置換
- smooth scrollを無効化
- 情報はDOM状態・テキスト・アイコンで完全に理解できる

参考: [MDN prefers-reduced-motion](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/At-rules/%40media/prefers-reduced-motion)

## 13. Living Core Renderer — 全面再設計を許可

`OctopusCanvas` を改善すること自体を目的にしない。最も美しく、意味が分かり、軽く、アクセシブルなLiving Coreを作ることを目的にする。

### 13.1 技術選定の自由

次のどれを選んでもよい。

- Canvas 2Dの全面書き換え
- SVG + DOM
- WebGL / Three.js / React Three Fiber
- Canvas/WebGLの生物表現 + SVG/DOMの情報レイヤー
- 上記のレスポンシブ切替

推奨は、**生物的な発光・触手は描画レイヤー、レンズ名・状態・操作はDOM/SVGレイヤー**というハイブリッド。これなら未来感、可読性、操作性、スクリーンリーダー対応を両立しやすい。

Three.js等を採用する場合は、初期bundleをcode splitし、WebGL非対応・省電力・Reduced Motion時の静的SVG/DOMフォールバックを用意する。単に3Dにしただけで情報が分かりにくくなるなら採用しない。

### 13.2 意味として必ず残すもの

- 中央の1つのCore
- 8つの腕/レンズ
- 4つの対角軸
- nodes時は外向きの並列探索
- synth時は中央へ知見が戻り織られる
- deepening時は対角2本が張り合う
- resonating時は離れた2本が同期する
- 失敗・未起動・完了が見分けられる

形状、数式、カメラ、色の細部、props、レンダラーは自由に変えてよい。

### 13.3 品質要件

- コンテナの実寸へ完全追従
- Retinaで鮮明
- モバイルで過度に小さな触手や読めないラベルを作らない
- 画面外・バックグラウンドで停止
- idle時は省電力、処理中だけ滑らかに
- Reduced Motionでは静止/低運動版
- Reduced Transparencyでは不透明版
- 描画レイヤーを隠してもDOMだけで全状態が理解できる
- 8レンズの順序は `nodeDisplay.ts` または新しい単一Design Modelで一元管理
- レンダラーとアプリ状態を疎結合にし、純粋なViewModelを受け取る

例:

```ts
interface LivingCoreViewModel {
  phase: UiPhase;
  mode: "light" | "deep";
  lenses: Array<{
    id: string;
    label: string;
    axis: string;
    status: LensUiStatus;
    emphasis: "none" | "tension" | "resonance";
  }>;
  reducedMotion: boolean;
  variant: "onboarding" | "hero" | "working" | "compact";
}
```

## 14. UI観測イベント契約 — 必要なだけ拡張してよい

「実際に起きていること」を正確に見せるため、フロントだけで推測せず、Workersが持っているライフサイクル情報をイベントとして送る。新しいLLMコールは不要だが、イベント名・型・エンドポイントは最高のUXに合わせて再設計してよい。

### 14.1 推奨イベント

```ts
type TraceEvent =
  | {
      type: "request.accepted";
      requestId: string;
      seq: number;
      startedAt: number;
      requestedPlan: "light" | "deep";
    }
  | {
      type: "route.completed";
      requestId: string;
      seq: number;
      plan: "light" | "deep";
      domain: string;
      activeNodeIds: string[];
    }
  | {
      type: "nodes.started";
      requestId: string;
      seq: number;
      nodeIds: string[];
    }
  | {
      type: "node.completed";
      requestId: string;
      seq: number;
      node: NodeView;
      elapsedMs?: number;
    }
  | {
      type: "synthesis.started";
      requestId: string;
      seq: number;
      quorum: string;
    }
  | {
      type: "answer.delta";
      requestId: string;
      seq: number;
      text: string;
    }
  | {
      type: "verification.started";
      requestId: string;
      seq: number;
    }
  | {
      type: "request.completed";
      requestId: string;
      seq: number;
      payload: DonePayload;
    }
  | {
      type: "request.failed";
      requestId: string;
      seq: number;
      code: string;
      retryable: boolean;
      userMessage: string;
    };
```

既存SSE形式を拡張しても、新しい統一eventに移行してもよい。`requestId` と単調増加の `seq` を持たせ、キャンセル後の古いtokenや競合レスポンスを無視せず安全に判定する。

### 14.2 深化・共鳴も進行を見せる

現在の `/api/deepen` と `/api/resonate` は完了まで一括待機なので、必要ならストリーミング版を追加する。

- `/api/deepen/stream`: 対角腕開始 → 2腕完了 → 再統合開始 → answer delta → done
- `/api/resonate/stream`: 2腕接続 → 第三案生成開始 → answer delta → done

モデル呼び出し回数は増やさず、既存呼び出しの開始/完了と出力ストリームを可視化する。JSON版を残すか置換するかは、コードの単純さとテスト容易性で判断する。

### 14.3 Heartbeatと切断

長い無通信時間が生じる場合はcommentまたはheartbeatを送り、クライアント側で「固まった」のか「考え続けている」のかを判定できるようにする。UIにはheartbeat自体を表示しない。

Abort時はクライアント状態を即時cancelledへし、可能な範囲でサーバー側の処理も中断する。中断できなかった処理の結果は `requestId` で破棄する。

### 14.4 真実性

バックエンドが発火していない段階を、アニメーションの都合で先に表示しない。ノード全並列開始時は全てworkingでよいが、完了は実イベント到着時だけ。時間を埋めるための偽イベントは禁止する。

### 14.5 テスト

- event順序
- seq重複/逆転
- cancel後の遅延token
- nodeの順不同完了
- heartbeat
- stream途中切断
- deepening/resonatingの各段階
- done時の最終状態整合

分析品質・原価ログ・クォーラム・Verifierがイベント再設計で変質していないことも回帰テストする。

## 18. 依存関係 — 上限ではなく品質と計測で決める

依存を増やさないことを目標にしない。アクセシビリティ、状態の正しさ、モーション品質、保守性が上がるなら導入してよい。自前実装の方が不安定になる領域では、成熟したライブラリを優先する。

候補:

- Markdown: `react-markdown`, `remark-gfm`
- Accessible primitives: Radix UI / React Ariaの必要パッケージ
- Motion: `motion`（旧Framer Motion系）等
- Local persistence: `idb` 等
- State machine: XState（複雑性が実際に下がる場合）
- Renderer: `three`, `@react-three/fiber`, `@react-three/drei`（採用判断を満たす場合）
- Testing: `@testing-library/react`, `@testing-library/user-event`, `@playwright/test`, `@axe-core/playwright`

すべてを入れる必要はない。判断基準は次の順。

1. ユーザー体験とアクセシビリティ
2. 状態・操作の正しさ
3. モバイル性能
4. 保守性
5. bundleサイズ

大きな依存はroute/component単位でcode splitする。完了報告では、追加した依存、採用理由、gzip bundle差分、低性能端末向けfallbackを示す。依存を避けるために品質を落とすことも、未来感のためだけに重い依存を積むことも避ける。

## 19. パフォーマンス目標

### 必須確認

- 390px幅のモバイルで処理中に操作が固まらない
- idle CanvasがCPUを浪費し続けない
- バックグラウンドタブでRAF停止
- token stream中も入力・スクロールが滑らか
- レイアウトシフトで本文位置が大きく跳ねない
- コンソールエラー/React warningなし

### 目標値

- INP相当: 200ms未満を目標
- CLS: 0.1未満を目標
- 処理中: デスクトップ50fps以上、一般的なiPhoneで30fps以上を目標
- idle: 描画頻度を落とし、発熱が気にならないこと

数字を達成できない場合は、装飾を守るより操作性を優先する。

## 22. 受け入れ条件

### 起きていることの理解

- [ ] `routing / nodes / synth / verify` が別フェーズとして画面に見える

- [ ] 各レンズのworking/done/失敗/未起動が見える
- [ ] 失敗ノードを隠さず、回答への影響も説明する
- [ ] 生のChain of Thoughtは表示しない

### UI

- [ ] タコが全面背景ではなく機能的なLiving Coreになっている
- [ ] 回答本文が最も読みやすく、最も強い階層にある
- [ ] TENSIONとRESONANCEの意味が色・形・文言で区別できる
- [ ] メタ情報は段階的開示になっている
- [ ] 初回Heroで価値が5秒以内に分かる

### UX

- [ ] 複数行入力・IME・Enter/Shift+Enterが正しく動く
- [ ] 実行停止・再試行ができ、入力と途中回答を失わない
- [ ] 過去回答を読んでいる人を自動スクロールで奪わない
- [ ] copy/share/retryが動く
- [ ] エラー時に次の行動が分かる
- [ ] 初回オンボーディングで8腕・4軸・中央脳を体験的に理解できる
- [ ] 会話を閉じて再度開いても履歴・summary・完了した思考詳細が復元される
- [ ] 新規会話、名前変更、削除、ドラフト復元が動く
- [ ] 設定からMotionとLiving Core表示を調整できる
- [ ] オフラインでも履歴とドラフトを失わない

### Mobile

- [ ] 390×844で重なり・横スクロール・切れがない
- [ ] 320px Reflowで主要機能を失わない
- [ ] ComposerがSafe Areaとキーボードに対応
