# 🐙 OctoBrain ロードマップ — P3完了から App Store 公開まで

P0〜P3(動くコア+ベンチ)完了後の全行程。各フェーズに入る前に、
この文書の該当節をもとに `docs/phases/P*_*.md` の詳細指示書を作成してから
Claude Code に渡す — これまでと同じ運用を続ける。

## 全体像

```
M1「動く」        P0 骨格 → P1 パイプライン → P2 フロント → P3 ベンチ   ✅(完了前提)
M2「壊れない」    P4 チューニング → P5 堅牢化
M3「アプリになる」 P6 アプリUX → P7 Capacitor/iOS
M4「売れる形」    P8 課金(IAP)
M5「世に出る」    P9 申請準備 → P10 リリース・運用
```

### この順番の理由

- **P4が最初**: P3の数字を放置しない。モデル最終選定・料金設計・改善はすべてベンチ結果が根拠になる
- **堅牢化(P5)がUX(P6)より先**: クォータや連打対策のないままUIを磨くと、公開直前に配管をやり直す羽目になる
- **課金(P8)がネイティブ(P7)の後**: IAPは実機とApp Store Connectに強く結びつき、手戻りが最も高くつく。土台が固まってから
- **申請(P9)は独立フェーズ**: AIチャットアプリは審査で見られる点が通常アプリより多い。片手間にしない

## P4 — チューニング: ベンチ結果の回収 【重さ: 中】

**目的**: P3の数字を製品品質と料金設計に変換する。

- `bench/results.md` を読み、**モデルの最終選定**(`models.ts` の SET_ME を実測に基づき確定。最新価格を必ず再確認)
- `NODE_MODEL_POOL` に2〜3社の軽量モデルを設定し、ノードの出力相関を下げる → 再ベンチで多角性スコアの変化を確認
- 弱かったカテゴリのプロンプト改良(ノード動詞の言い換え / 統合手続きの調整)。**1変更→再ベンチ**のループ。questions.json は変えない(定点観測のため)
- **境界の正直さ**を実装: 数学・最新情報など苦手系をRouterで検出したら、回答冒頭で「この種の問いは概算しかできない/最新情報は持っていない」と正直に言う挙動。誤魔化さないことを製品の信頼性にする
- 実測原価から料金設計の下書き: 1回答あたり実コスト × 想定利用回数 → 無料枠の損益分岐と有料価格の根拠
- 統合脳の品質が不足していた場合の分岐は末尾「リスクと分岐」参照

**完了の目安**: models.ts に SET_ME が残っていない / 再ベンチで初回比の改善が数字で見える / 料金設計メモ(`docs/pricing_notes.md`)がある

## P5 — 堅牢化: 公開に耐える配管 【重さ: 中】

**目的**: 不特定多数に触られても壊れない・赤字にならない状態。

- **クォータの実ブロック**: 月間上限超過で 429 + UIメッセージ(P1では読むだけだった)
- **連打防止**: 同一clientIdの同時実行1本制限 + 最小リクエスト間隔(KVで軽く)
- **全体タイムアウト予算**: リクエスト全体で上限(例30秒)。※Workers無料プランのCPU時間制限は「fetch待機はCPU消費しない」ため8並列でも問題ないが、指示書に明記して意識させる
- 入力バリデーションの再点検(input 4000字 / summary 500字は実装済み → 異常系テスト追加)
- summary改ざんの整理: クライアント経由だが、悪用しても本人の回答品質が下がるだけ(リスク低)と設計メモに明記
- **運用スクリプト** `ops/report.ts`: KVの原価ログから日次コスト・失敗率・平均msを表で出す(P10の監視で毎日使う)

**完了の目安**: 連打してもコストが線形に増えない / 上限超過の挙動がUIまで通っている / `npm run ops:report` が動く

## P6 — アプリUX: 道具から製品へ 【重さ: 中〜重】

**目的**: 毎日開きたくなるアプリの形にする。

- **会話の永続化**(Capacitor Preferences で開始、重くなればSQLite): 会話一覧・タイトル自動生成(compressノード流用で安く)・削除・新規会話(ローリング要約もリセット)
- **オンボーディング3枚**: 「8本の腕が別々に考え、頭が統合する」世界観を最初に見せる。タコアニメーションを主役に
- **設定画面**: モード選択(おまかせ=auto / じっくり=complex固定 ← プレミアム布石)、今月の利用回数表示(クォータAPI)
- 回答のコピー / 共有
- 空状態・エラー状態・オフライン時のデザイン
- アプリアイコン & スプラッシュ(タコ。世界観の顔になる)
- 「8つの視点を見る」の磨き込み: どのノードが統合に強く効いたかの表示は面白いが任意

**完了の目安**: アプリを閉じて開いても会話が残る / 初見の人がオンボーディングだけで世界観を理解できる

## P7 — Capacitor化: iOSネイティブ 【重さ: 中】(ねこ電卓等の経験が直接活きる)

**目的**: iPhone実機で気持ちよく動く。

- Capacitor 6 導入、iOSプロジェクト生成(`web/` はP2でCapacitor互換にしてあるので薄い作業のはず)
- **Safe Area対応**(特に下部の入力フッター)、キーボード出現時のレイアウト(Keyboardプラグイン)
- **OctopusCanvasのdevicePixelRatio対応**: Retinaでぼやけないように。※「描画ロジック挙動変更禁止」ルールとの整合 → 解像度対応は忠実度向上として許可する旨をP7指示書に明記すること
- 実機での確認: fps・発熱・バッテリー(Canvasアニメが常時走るため。バックグラウンド時にrequestAnimationFrame停止の確認)
- ステータスバー配色、ダークUI前提の調整
- API向き先: `VITE_API_BASE` を本番Workers URLへ。ALLOWED_ORIGINにcapacitor://localhost系を追加
- TestFlight内部テストで自分の端末に配布

**完了の目安**: 実機で1日普通に使って違和感がない / 発熱・電池が気にならない

## P8 — 課金: IAP(RevenueCat) 【重さ: 重】(Zanshinの設計知見を流用)

**目的**: 続けられる収益構造。Apple 3.1.1(デジタル機能はIAP必須)完全準拠。

- **プラン設計**(P4の原価が根拠):
  - Free: 月N回、モードはauto(simple/normal中心)
  - Pro(月額サブスク): 回数大幅増 + complex常時(8基フル)+ 統合脳を上位モデルに(models.tsにsynth_premiumロールを追加)
- RevenueCat導入、ペイウォールUI、購入復元、解約導線
- **サーバー側のプラン検証**: 段階設計でよい。
  - 第一段階: クライアントがエンタイトルメントを申告 → サーバーはクォータで防御(不正しても回数上限まで)
  - 第二段階(ユーザーが増えたら): RevenueCat APIでサーバー検証
- 価格表示の地域対応はApp Store Connectに任せる
- 審査対策: 課金前に無料で価値が体験できる導線を必ず残す

**完了の目安**: サンドボックスで購入→Pro機能解放→復元まで通る / 未課金でもアプリが完結して使える

## P9 — 申請準備: AIアプリ特有の審査対策 【重さ: 中】

**目的**: 一発通過を狙う。AIチャットは通常アプリより見られる点が多い。

- **プライバシーポリシー**: 入力テキストが処理のため外部LLM APIへ送信されること / KVログの保持期間(90日)/ トラッキングなし、を明記
- **利用規約**: AI出力の正確性の免責、禁止利用
- **App Privacy申告**: 「ユーザーコンテンツ(入力)」収集あり・トラッキングなし
- **年齢レーティング**: 自由入力のAIチャットのため **17+ が安全**(無制限Web検索なし・Verifierありでも、生成AIは保守的に)
- **ガイドライン1.2(UGC類似)対応**: 不適切な出力を報告する導線(設定内のメールリンクで可)+ Verifierによる出力チェックの存在を審査ノートに記載
- **4.2(最小機能)対策**: 「単なるLLMラッパー」と見られないよう、審査ノートに独自性を明記 — 8ノード並列アーキテクチャ、視点の可視化UI、ベンチ結果
- **3.1.1**: 課金は全てIAP。外部購入への誘導文言・リンクがないこと最終確認
- ストア素材: スクショ5枚(①タコ+回答 ②8つの視点展開 ③処理中アニメ ④オンボーディング ⑤料金)、紹介文はP3の数字を使う(例:「8つの視点で考える思考エンジン」)。「最強AI」系の誇大表現は避ける
- ログイン不要なのでデモアカウント不要。その旨も審査ノートへ

**完了の目安**: App Store Connect のすべての項目が埋まり、審査ノートが書けている

## P10 — リリースと運用 【重さ: 軽〜継続】

- TestFlight外部テスト(小規模でよい。マダミス仲間・音楽仲間に触ってもらうのは良いテスト)

## 横断事項(全フェーズ共通)

- **コスト意識**: 新しいモデル呼び出しを追加するときは必ずcallModel経由(絶対ルール5)。原価ログに乗らない呼び出しをゼロに保つ
- **1フェーズ=1コミット(PR)** の運用継続。各フェーズ前に詳細指示書を作る
- **ベンチは資産**: 大きな変更(モデル・プロンプト)の後は `npm run bench` で定点観測。questions.json は変えない
- **ドキュメント同期**: 設計変更したら `00_architecture.md` を必ず更新(唯一の正を保つ)

## リスクと分岐

**P3の結果が期待外れだった場合(品質が単発上位に大きく負ける)** — 3ルート:
1. 統合脳を1ランク上のモデルへ(コスト増は小さい。統合は1回だから) ← まず試す
2. ノード数を絞って質の高い4基構成をデフォルトに(弱いノードを混ぜると下がる、はMoA研究の知見どおり)
3. プロンプト改良ループ(動詞の具体化・統合手続きの強化)
「コスト1/4で品質90%」でも製品としては十分勝ち。完勝を狙わない。

**審査リジェクトの典型と対応**:
- 4.2(最小機能)→ 審査ノートで独自アーキテクチャを説明、8視点UIのスクショ添付
- 1.2(生成コンテンツ)→ 報告導線とVerifierの説明を追記して再提出
- 17+への変更要求 → 素直に従う(争わない)

**コスト暴走**: クォータ(P5)が第一防壁。第二防壁として日次コスト上限の全体ブレーカー(超過したら全リクエストをsimpleモードに落とす)をP5で仕込んでもよい

**Workers/KVの無料枠超過**: 初期は余裕(10万req/日)。超えたら有料プラン($5/月〜)へ — それは嬉しい悲鳴
